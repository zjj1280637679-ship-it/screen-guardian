Screen Guardian 项目结构实验稿

目标
====

本实验不替换普通 GitHub 项目结构，而是在现有工程结构上叠加一套 AI-first 知识结构。

普通工程结构继续负责安装、测试、审阅、发布和运行：

- README.md
- docs/
- mcp/
- scripts/
- skills/
- tests 或 smoke/evaluation scripts
- package.json
- LICENSE

新增知识结构负责让人类维护者、Codex、子代理和未来审计者更快理解项目：

- docs/whitepaper/
- scenario-cards/
- reference-source/
- optimized-runtime/
- traceability/

其中前四个目录是知识结构的四层，traceability/ 不是第五层，而是把四层、测试、运行机制和回执字段连起来的关系层。

核心判断
========

这个项目不应该只有“能跑的代码”和“用户文档”。Screen Guardian 的复杂度来自 AI 如何理解桌面、如何选择感知路径、如何判断截图是否可信、如何控制上下文和副作用。

因此项目需要两套入口：

1. GitHub 工程入口
   面向安装、运行、测试、发布和外部审阅。

2. AI-first 认知入口
   面向理解、推理、场景覆盖、参考实现和可追踪验证。

这两套入口并列存在，但权责不同。

四层知识结构
============

1. 白皮书层
-----------

路径：

docs/whitepaper/

作用：

白皮书是项目的总知识源。它不是单纯宣传文档，而是把项目理念、使用方式、设计原则、验证标准和安全边界组织成章节。

建议章节：

- philosophy：为什么需要兼容优先的本地 AI 感知基础设施。
- usage：AI 和用户应该如何从直觉任务进入工具。
- design：adapter、guard、receipt、route、budget、workflow envelope 如何配合。
- verification：哪些条件证明功能没有偏离设计。
- safety：反滥用、授权边界、本地默认、无隐藏上传和无隐藏调度。

现有 docs/WHITEPAPER.md 仍然保留，作为完整英文白皮书；docs/whitepaper/ 负责把它拆成更容易引用和追踪的章节入口。

2. 场景卡森林
-------------

路径：

scenario-cards/

作用：

场景卡是白皮书原则在真实世界中的实例化。每张卡描述一个用户、一个触发条件、一条推荐路径、风险、失败分支和验收条件。

场景卡不等同于教程，也不等同于测试。它介于需求、产品设计、测试计划和 AI 行动模板之间。

典型场景卡包括：

- 快速看屏幕。
- 窗口静默捕获。
- 未渲染保护。
- 只保存文件不进入上下文。
- 短时间变化观察。
- 网页长截图。
- 新手用户排障。
- 高强度用户批量窗口报告。

场景卡森林的价值是把“工具能做什么”改写成“什么人、什么情况下、为什么这样做”。

3. 高标注扁平化参考源
----------------------

路径：

reference-source/

作用：

参考源代码是给人和 AI 读的，不是给机器最快运行的。它应该少跳转、强注释、显式状态机、显式边界和显式失败分支。

它的目标是成为“可执行白皮书”：

- 每个工具入口能对应到白皮书概念。
- 每个 guard 能解释为什么拦截、等待、保存或退回决策菜单。
- 每个副作用都有清楚注释。
- 每个预算、feature flag、runtime limit 都能看出设计原因。

当前版本先建立目录和映射说明，不立即复制或重写运行时代码。后续可以逐步抽取高标注参考实现。

4. 编译器化压缩高效运行代码
----------------------------

路径：

optimized-runtime/

现有运行代码位置：

- mcp/server.cjs
- scripts/screen_guardian_capture.py

作用：

运行代码负责稳定、快速、低依赖地完成任务。它可以少注释、少冗余、优化路径，但必须能追溯到参考源和白皮书概念。

理想关系是：

白皮书定义原则
  -> 场景卡检验真实需求
  -> 高标注参考源表达可读逻辑
  -> 高效运行代码承担实际运行
  -> traceability 反向证明没有偏离原则

当前版本不把运行代码搬到 optimized-runtime/，因为 MCP、插件缓存和文档链接都依赖原路径。optimized-runtime/ 先作为“运行产物层说明”和未来编译映射入口。

追踪关系层
==========

路径：

traceability/

作用：

追踪关系不是第五层知识结构。它把四层连起来，避免白皮书、场景卡、参考源和运行代码各说各话。

建议映射：

- 白皮书章节 -> 场景卡
- 场景卡 -> 验收条件
- 验收条件 -> 测试或 smoke 路径
- 参考源概念 -> 运行时代码位置
- 运行回执字段 -> 白皮书中的机器可读回执概念

审阅方法
========

为了降低等待时间和避免子代理目标漂移，复杂问题默认拆成短课题。

每个课题至少说明：

- 课题：只检查什么。
- 目标：这次判断要证明或排除什么。
- 非目标：这次不解决什么。
- 输出格式：只要缺口、风险、文件名、验收条件，还是需要建议补丁。
- 最小验证动作：下一步最小能降低不确定性的检查。

推荐拆法：

- 结构课题：白皮书、场景卡、参考源、运行映射、追踪关系是否各自承担正确职责。
- 目标错位课题：某个文件是否在解决和当前目标无关的问题。
- 覆盖课题：某个场景、风险或验收条件是否缺失。
- 验证课题：编码检查、合同验证、smoke 测试是否扫到了新增结构。
- 因果课题：某个现象只是共现、可能原因、贡献原因，还是必要或充分条件。

子代理默认只承担窄任务，不承担最终结论。超时的子代理结果降级为“未取证”，主代理继续用本地证据推进，并把问题重新拆小。

最小落地方式
============

本次实验只做结构，不做大规模重构：

1. 新增 docs/whitepaper/ 章节入口。
2. 新增 scenario-cards/ 场景卡森林入口和少量核心卡。
3. 新增 reference-source/ 参考源说明。
4. 新增 optimized-runtime/ 运行代码映射说明。
5. 新增 traceability/ 映射说明和机器可读草稿。
6. 更新 README，让普通 GitHub 用户知道这些目录是认知层，不是替代工程结构。
7. 更新验证脚本，让这些文件进入 UTF-8 和设计覆盖检查。

边界
====

- 不移动 mcp/ 和 scripts/ 里的运行时代码。
- 不改变 MCP 工具 schema。
- 不改变默认工具面。
- 不改变安装和发布流程。
- 不把参考源和运行源强行拆成两套逻辑。
- 不承诺当前已经有完整编译器；这里只是先建立未来可以演进的结构。
- reference-source/ 和 optimized-runtime/ 当前是结构脚手架和映射入口，不表示已经存在完整参考编译器或自动压缩流水线。

最终定义
========

Screen Guardian 保留普通 GitHub 项目的工程结构，用于安装、测试、审阅和发布；同时叠加 AI-first 知识结构，用于理念表达、场景覆盖、参考实现和可追踪验证。

这让项目既能符合开源项目的普遍发布流程，也能保持我们想要的审美：不是一堆工具堆叠，而是一套能被 AI 和人类共同理解的本地感知系统。
